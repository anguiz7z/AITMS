Docs

Reference content that used to live as separate top-level nav items. Each card opens the original page — same content, same routes, just collapsed off the global nav. Flip the matching feature flag in src/atms/features.py (or set the env var) to restore an individual page to the top-level nav. See README "Re-enabling hibernated features".

Knowledge base →

Bundled corpus counts: playbooks, frameworks, compliance controls, MITRE ATLAS, CISA KEV / EPSS, cloud-service catalog. The data driving every analysis.

Playbooks →

One per component type (121 total). Each lists the typical AI-specific threats, pre-mapped to STRIDE-for-AI, OWASP LLM Top 10, and MITRE ATLAS. Pure YAML in kb/playbooks/.

MAESTRO →

The 7-layer agentic-AI threat-modelling reference architecture from CSA (2025). Browse per-layer threats — Foundation Models, Data Operations, Agent Frameworks, Deployment, Evaluation, Security & Compliance, Agent Ecosystem.

OWASP Agentic →

OWASP Agentic AI Threats & Mitigations (OWASP ASI, 2025). 15 canonical threats (T1–T15) plus 2 ATMS extensions — memory poisoning, tool misuse, intent breaking, rogue agents, inter-agent protocol abuse.

Methodology →

Per-threat framework provenance. STRIDE-for-AI category, MITRE ATLAS technique IDs, OWASP LLM / Agentic refs, NIST citations — all rendered side-by-side for the last analysed system.

Architecture →

Live mermaid diagram of every ATMS module — inputs, engines, KB, reporting, surfaces. Drift-guarded by scripts/check_architecture_drift.py.

Capabilities →

One-page surface inventory: every input format, framework, export, and HTTP endpoint exposed by this build. Live counts via /api/v1/metrics.

About →

What ATMS is, why STRIDE-for-AI is the right anchor, how risk scoring works, license, source repo.

Tools

Working analysis surfaces beyond the core Analyze loop. These are all enabled — they live here (rather than as separate top-bar tabs) to keep the global nav focused. Pin any one back onto the top bar with its ATMS_FEATURE_NAV_<NAME>=1 flag.

IaC ingest →

Generate a System model from Terraform, CloudFormation, Kubernetes, Azure ARM/Bicep, Pulumi, or docker-compose. Deterministic, offline.

Evidence →

Enrich a threat model with VAPT / scan findings (Nessus, SARIF, CSV, STIX 2.1) so evidence-backed threats drive the final severity.

Red-team →

Ingest red-team / breach-and-attack-simulation results to mark threats as observed or exploited in the report.

Compliance →

Map threats + mitigations onto compliance frameworks (NIST, ISO, CSA, etc.) and export the coverage matrix.

Devices →

Browse the OT / IoT / network device catalog ATMS uses to classify components and seed device-specific threats.

Diff →

Compare two saved analyses to see which threats / mitigations changed — useful for quarterly re-assessments.
